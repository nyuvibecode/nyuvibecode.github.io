üß≠ Vibe Coding Rubric & Scoring Policy

üîç Rubric Scope & Policy

  • Projects will be evaluated by a human judging panel drawn from industry, academia, and innovation sectors, with AI-assisted reference for consistency.
  • Judgments are human-led, not AI-only or AI-dominated.
  • Teams will receive qualitative feedback and rubric-based summaries.
  • Final numeric scores will not be publicly disclosed, but relative rankings will appear on the leaderboard.

üß© Scoring Categories (100 Points Total)

Projects are evaluated on five dimensions, each weighted equally at 20 points.

Category Description Key Considerations
Functionality (20 pts) How well the project works. Feature completeness, correctness, reliability, error handling, innovation, and alignment between documentation and demo performance.
UI/UX Design (20 pts) How users interact and feel. Clarity of interface, flow, accessibility, responsiveness, and polish. A clean, intuitive design earns higher marks.
Impact (20 pts) Why it matters. Significance of the problem solved, creativity, real-world or academic relevance, scalability, and potential for adoption or further development.
Prompt Quality (20 pts) How well the AI is used. Prompt clarity, reproducibility, safety/guardrails, adaptability, and engineering craftsmanship. Demonstrate thoughtful LLM utilization.
Readability (20 pts) How easy it is to understand and maintain. Code clarity, modularity, documentation completeness, demo instructions, and README organization. Professionalism counts.

üß† Judging Considerations

  • Judges will read the README/documentation and watch the demo video before evaluating the live demo.
  • LLM disclosure is mandatory — clarity about model choice and use will influence the Prompt Quality score.
  • Extra features are encouraged, but only reliable, working features count.
  • Evaluation combines documented evidence (readme, video, code) and observed performance (demo).
